# 登出后重新登录相机崩溃修复总结

## 问题现象

用户从 admin 登出 → opera 重新登录后，程序抛出未处理异常：

```
SapNativePointerException: Invalid native class pointer.
The Dispose method may already have been called.
   在 XferTrashNotifyCallback(SapXferCallbackInfo* info)
```

第一眼看起来连接成功了，但异常已经发生，潜在风险很大。

## 问题根因

### 对比 disconnect() 和 cleanup()

| 步骤 | `disconnect()`（正常切换，正常） | `cleanup()`（登出，崩溃） |
|------|--------------------------------|--------------------------|
| 停止采集 | Freeze → sleep(0.1s) → Abort | Freeze → sleep(0.05s) → Abort |
| 中间操作 | 无 | 恢复触发器、帧率、曝光、LUT（触发 SDK 内部通信） |
| 解绑回调 | `xfer.XferNotify -= callback` | **缺失** |
| 销毁等待 | sleep(0.1s) | **缺失** |
| xfer 销毁 | `Destroy()` → `Dispose()` | **仅 `Dispose()`，无 `Destroy()`** |
| 错误隔离 | 每个 Destroy/Dispose 独立 try | Destroy + Dispose 同 try，一个失败另一个跳过 |

### 三个根本原因

**1. 没解绑 `XferNotify` 回调（主因）**

`SapBufferWithTrash` 在创建时 SDK 内部自动注册了 `XferTrashNotifyCallback`。`cleanup()` 销毁 xfer 前没有调用 `xfer.XferNotify -= self._on_frame_callback` 来解绑。

旧回调残留在 SDK 内部回调队列中，新登录创建连接后触发 → 回调里调用 `SapHelper.FindTransfer()` 查找旧 xfer → 旧 xfer native 指针已释放 → 抛出 `SapNativePointerException`。

**2. xfer 只调了 `Dispose()` 没调 `Destroy()`**

Sapera SDK 规范要求：
- `Destroy()` — 释放非托管资源（native 指针、采集线程、缓冲区）  
- `Dispose()` — 释放 .NET 托管包装对象

直接 `Dispose()` 跳过非托管资源释放，native 指针变成悬空指针，但 SDK 内部回调队列还持有它的引用。

**3. Freeze 后等待时间不足（0.05s）**

Freeze 只是请求暂停，SDK 内部可能还有正在传输的帧和待执行的回调。0.05s 太短，回调队列没排空就销毁对象。

## 修复方案

### 修改文件

`InspectMainWindow.py` — `CameraController.cleanup()` 方法（第1373行附近）

### 改动1：延长采集停止后的等待时间

```python
# 修复前
self.xfer.Freeze()
time.sleep(0.05)   # 50ms，太短

# 修复后
self.xfer.Freeze()
time.sleep(0.5)    # 500ms，足够 SDK 内部回调队列排空
self.xfer.Abort()
time.sleep(0.1)    # Abort 后再给 100ms 缓冲
```

### 改动2：销毁前解绑传输回调

```python
# 新增（修复前不存在）
if self.xfer:
    try:
        self.xfer.XferNotify -= self._on_frame_callback
    except Exception:
        pass
```

### 改动3：xfer 先 Destroy 再 Dispose

```python
# 修复前
self.xfer.Dispose()   # 只 Dispose，没 Destroy

# 修复后
try:
    self.xfer.Destroy()      # 先释放非托管资源
except Exception:
    pass
try:
    self.xfer.Dispose()      # 再释放托管包装
except Exception:
    pass
```

### 改动4：Destroy 和 Dispose 分开 try-except

```python
# 修复前（buffers / acq_device）
try:
    self.buffers.Destroy()
    self.buffers.Dispose()   # 如果 Destroy 抛异常，Dispose 被跳过
except Exception:
    pass

# 修复后
try:
    self.buffers.Destroy()
except Exception:
    pass
try:
    self.buffers.Dispose()
except Exception:
    pass
```

### 完整修复后的 cleanup() 流程

```
1. Freeze() → sleep(0.5s) → Abort() → sleep(0.1s)
       ↓  确保 SDK 内部回调队列排空
2. 恢复默认设置（触发模式、帧率、曝光、LUT）
       ↓
3. xfer.XferNotify -= self._on_frame_callback   ← ★ 解绑回调
       ↓
4. xfer.Destroy() → xfer.Dispose() → xfer = None
       ↓  先非托管，再托管
5. buffers.Destroy() → buffers.Dispose() → buffers = None
       ↓  分开 try-except
6. acq_device.Destroy() → acq_device.Dispose() → acq_device = None
       ↓  分开 try-except
7. location = None, is_running = False
```

---

## 为什么 disconnect() 没问题而 cleanup() 有问题

两个方法做同样的事（释放相机资源），但实现不一致：

- `disconnect()` 是后来重写的，遵循了正确的 Sapera SDK 生命周期管理
- `cleanup()` 是更早写的，沿用了旧代码的简化写法

这次修复让 `cleanup()` 和 `disconnect()` 的销毁流程保持一致。
